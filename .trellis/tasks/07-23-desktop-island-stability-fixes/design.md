# 技术设计：桌面灵动岛稳定性修复

## 边界

- `IslandController` 负责形态请求的取消、串行化和销毁保护。
- `QingyaWindowController` 负责显示器选择、窗口属性回滚和主窗/岛模式转换。
- `island_bar.dart` 只保留现有视觉动画，不承载新的窗口生命周期逻辑。

## 方案

1. 为 morph 和窗口形状同步分别维护递增版本号，所有 `await` 返回后检查版本、启用状态和当前窗口模式。
2. 将 `_syncWindowShape` 收敛为单一异步队列（或等价的 latest-wins 调度器），只允许最新目标写入 `_appliedWinW/H`。
3. 进入岛时根据保存的主窗矩形选择显示器，优先使用包含主窗中心的显示器；无保存矩形时使用光标显示器，再回退主显示器。
4. 进入岛失败时执行最小的 normal chrome（主窗外观属性）恢复，并保持明确的 `hidden`/`normal` 模式语义。
5. `dispose`、禁用和重置路径统一取消计时器、清空队列并使所有版本失效。

## 数据流

`交互/状态更新 -> requestForm -> 最新目标形态 -> 窗口尺寸同步 -> UI phase`

展开时先确保 HWND 尺寸足够，再更新 UI；收缩时先更新 UI，动画结束后只由当前版本缩小 HWND。

## 兼容性与回滚

- 保持现有常量、`DesktopWindowMode` 和单窗约束不变。
- 失败时可回滚本任务涉及的四个桌面文件，不触碰会话或服务端代码。
- 不新增第三方依赖。

## 风险

- Windows 窗口 API 调用可能在模式切换时返回延迟结果，因此版本检查必须放在每个关键 `await` 之后。
- 多显示器坐标可能包含负值，选择显示器时必须使用实际屏幕矩形而不是仅使用宽高。
