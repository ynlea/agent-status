# 桌面灵动岛稳定性修复

## Goal

修复 Windows 桌面灵动岛在快速交互、主窗与岛切换以及多显示器场景下的状态竞态，确保最终形态、窗口尺寸和所在显示器一致。

## Confirmed Facts

- 当前采用单窗方案，主窗和灵动岛通过 `DesktopWindowMode` 切换。
- 主窗几何插值已移除；岛内形态由异步 morph（形态切换）和窗口尺寸同步共同驱动。
- 现有实现存在未失效的延迟 morph、并发窗口尺寸同步，以及进入岛时固定使用主显示器的问题。
- 当前测试覆盖视图模型，缺少窗口生命周期和岛控制器竞态测试。

## Requirements

- 任何主窗恢复、重新进岛、禁用灵动岛或控制器销毁操作，都必须使此前未完成的形态切换和延迟窗口调整失效。
- 窗口尺寸同步必须串行或带版本校验；旧请求不得覆盖更新的目标尺寸、模式或已应用尺寸记录。
- 灵动岛应定位在主窗所在的显示器；无法确定时才回退到当前光标显示器或主显示器。
- 进入灵动岛失败时，窗口应恢复到可恢复主窗所需的 chrome（窗口外观与行为属性），不能留下透明、置顶或不可调整大小的残留状态。
- 不改变会话过滤、通知文案、通知开关和单窗架构。
- 补充可自动执行的控制器/窗口状态测试；Windows 实机验证作为发布前检查项。

## Acceptance Criteria

- [ ] 连续悬停、离开、点击列表、通知打断后，最终 phase（形态）和 HWND（窗口句柄）尺寸与最后一次请求一致。
- [ ] 主窗恢复或重新进岛后，不会出现旧延迟任务再次改变岛形态或窗口尺寸。
- [ ] 控制器销毁后，已排队的异步任务不会访问或更新已销毁状态。
- [ ] 副显示器上的主窗进入灵动岛时，岛仍出现在同一显示器的顶部中心。
- [ ] 进入灵动岛任一步骤失败后，下一次恢复主窗可正常显示、交互和调整大小。
- [ ] 相关 Dart 静态检查和新增测试通过；不引入异常日志风暴。

## Out Of Scope

- 重新引入双窗、第二 Flutter Engine（Flutter 引擎）或原生黑块岛窗。
- 改动 macOS/Linux 桌面壳、会话业务逻辑或播报内容规则。
- 本任务不负责提交或归档父任务已有的未完成改动。
